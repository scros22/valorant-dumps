# valorant-dumps

Periodic offset dumps for `VALORANT-Win64-Shipping.exe`, generated by an
internal kernel-mode page-walking dumper running an offline pattern
scanner against the live game image.

This repository ships **only the resolved RVAs** â€” no binaries, no SDK
source, no game code. The patterns this is based on come from the
[ProtonDev-sys/valorant-offset-dumper](https://github.com/ProtonDev-sys/valorant-offset-dumper)
community signature set, with extra patch-resilient fallbacks layered on
top so we don't lose offsets to small instruction-encoding changes.

## What's here

```
offsets/  valorant-offsets-YYYY-MM-DD.txt   <- one snapshot per Riot patch
runs/     dump-run-YYYY-MM-DD.md            <- environment / coverage / notes for that run
docs/     research specs + IDA MCP string tables (no binaries)
```

Each file is self-contained and dated. Pick the one that matches the
build of VALORANT you're working with. For **sig vs struct discipline**, full
IDA `find_regex` synthesis, and reference JSON, start at
[`docs/SDK_DIRECTORY_INDEX.md`](docs/SDK_DIRECTORY_INDEX.md).

## Offset format

```
// VALORANT offset dump â€” generated by vdumper
// Source: kernel-mode PE dump + ProtonDev sigs + community fallbacks

ProcessEvent: 0x1B74230
StaticFindObject: 0x1BA05C0
GWorld: 0xC1AD2A0
FNamePool: 0xC34B800
...
```

Values are **RVAs** (offsets from the module base). To get a live VA:

```
LiveVA = base_of_VALORANT-Win64-Shipping.exe + RVA
```

In the 2026-04-26 capture the module base was `0x7FF7223B0000`, so e.g.
`ProcessEvent` lived at `0x7FF723F24230`. Module base randomizes per
launch (ASLR) â€” always read it from `PEB->Ldr` at runtime, never hard-code.

## How values are produced

1. A kernel driver lets the dumper page-walk the live VALORANT process
   and read its executable sections without ever calling `OpenProcess`,
   touching its handle, or running code inside it.
2. The dumper writes a copy of the PE image to disk (degraded â€” Vanguard
   blocks ~30-40% of `.text` pages, which is fine for sig matching but
   unsuitable for full reverse engineering).
3. An offline pattern scanner (section-aware, supports DirectRva and
   RIP-relative-32 resolution) runs against that image with the
   ProtonDev signature set + multi-pattern fallbacks for offsets known
   to drift across patches (notably `FNamePool`).
4. Resolved RVAs are written to the dated `.txt` file in this repo.

The live game is **not modified** at any step â€” no IAT hijacking, no
shellcode, no DLL injection, no Vanguard contact beyond passive
read-only memory probes from kernel context.

## Patch cadence

Dumps are produced opportunistically when a new VALORANT patch is
released or when an offset visibly stops working in downstream tooling.
There's no SLA; check the date on the `.txt` you're reading.

## Refining sigs

If a symbol shows up under `// Missing` in the latest `.txt`, the pattern
likely needs a new fallback variant. The signature definitions live in
the dumper source (not in this repo) under
`src/research/valorant_automation.hpp` (`kTier1` for stable, `kTier2`
for multi-pattern groups). PRs against the offsets here aren't a
meaningful workflow â€” open an issue describing which symbol is missing
on which build instead.
